# 3D 角色、人格设定与本地行为画像

这套系统现在已经把「模型资源」「角色 prompt」「用户本地行为画像」三件事接在一起了。

## 1. 现在有哪些组成

1. 3D 模型资源
   - 放在 `public/models`
   - 当前内置有 `baobao.glb` 和 `fanfan.glb`
   - 也支持用户上传自定义 `.glb/.gltf/.vrm`

2. 角色档案
   - 每个角色都可以配置：
   - 名字
   - 简介
   - `personalityPrompt`
   - `systemPrompt`
   - 绑定的 `modelPath`
   - 自动分析出的 `capabilities`

3. 本地行为画像
   - 记录用户切线程、切模型、切角色、开终端、要求明确结束态、查看完整错误等行为
   - 生成一份本地摘要
   - 只喂给 3D 角色的 avatar director

## 2. 模型能力分析现在是怎么来的

无论是内置模型还是上传模型，系统都会尽量拿到这些信息：

1. 动画数量
2. 动画名称
3. morph target 数量
4. morph target 名称
5. 推荐动作映射

这些能力会影响两件事：

1. 前端角色运行时怎么选动画
2. 泡泡导演在生成动作/语气时，知道这个模型“能做多少事”

例如：

- `baobao.glb` 动画较多，更适合丰富动作
- `fanfan.glb` 动画较少，所以提示词会更克制，不会暗示太夸张的表演

## 3. 角色弹窗里现在的流程

当你在设置里打开“编辑当前角色 / 新建自定义角色”时，流程是这样的：

1. 先选一个已有模型
   - 比如 `泡泡` 对应的 `baobao.glb`
   - 或 `饭饭` 对应的 `fanfan.glb`

2. 如果你有新的模型文件，也可以直接上传
   - 上传后服务端会分析模型动画和 morph 信息
   - 然后把能力摘要回填到角色表单

3. 你再填角色 prompt
   - 性格设定
   - 额外 system prompt

4. 保存后会持久化到本地 store
   - 角色配置不会因为切线程丢掉
   - 当前选中的角色也会被记住

## 4. 用户行为画像会怎么参与

本地画像不会替你做决定，它主要做“软偏好增强”。

例如它会总结：

1. 用户很在意明确结束态
2. 用户更喜欢看完整错误原文
3. 用户常切线程，多线程上下文要稳
4. 用户常开终端和 dev server
5. 用户很在意角色可配置性

这些总结会被注入给 avatar director。

主 `build-agent` 不读取这份画像，避免把执行 agent 和陪伴角色混在一起。

## 5. 现在如果你说“帮我启动开发服务器”

当前推荐流程是：

1. 主 agent 优先用 `exec_command`
2. 先等待一个短暂 `yield_time_ms`
3. 如果首屏还没出现真实地址，就继续用 `write_stdin(chars: "")` 轮询
4. 直到拿到真实 URL、真实报错，或者明确知道进程还在跑但尚未打印地址

这时候 3D 角色会同时拿到：

1. 当前线程上下文
2. 你的行为画像
3. 当前角色的人格设定
4. 当前模型的动作能力摘要

所以它的气泡文案和动作，不再只是“通用陪伴话术”，而是会更贴近你当前这个角色配置。

## 6. 后续还可以继续扩展什么

这套结构现在已经支持继续往下长：

1. 给不同模型做更精确的动作语义标注
2. 把表情能力映射进 bubble 风格和语气
3. 给不同角色建立不同的默认导演策略
4. 做更细的用户行为偏好分层，比如“喜欢详细解释 / 喜欢先改再说 / 喜欢看原始日志”
