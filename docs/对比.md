# Mastra / OpenCode / Codex Prompt 架构分层对比

## 这份文档怎么读

这次不再按“先讲完 Mastra，再讲 OpenCode，再讲 Codex”的方式写。

因为那样适合做单系统说明，不适合学习和比较。

这份文档改成按“层”来写。每一层都固定对照四件事：

1. Mastra 这一层怎么做
2. OpenCode 这一层怎么做
3. Codex 这一层怎么做
4. 这一层三者的关键差异是什么

这样你可以在同一个视角下直接看三套系统的不同设计。

## 一、总览：先把 Prompt 拆成哪几层

如果把这三套系统抽象来看，真正影响模型行为的大致有这些层：

1. 静态 prompt 层
2. 模型专属 prompt / 模板层
3. 运行时环境注入层
4. 工具规则与执行契约层
5. instruction / skills 注入层
6. memory / compaction / summary 层
7. mode / turn 控制层
8. route / orchestrator 组装层

这次我们先重点看第一层：**静态 prompt 层**。

后面的层我也一并按同样结构整理好了，这样你后面能继续一层一层看。

## 二、第一层：静态 Prompt 层

这一层的问题是：

- 系统最基础、最不依赖运行时状态的 prompt 文本是什么
- 它主要负责什么
- 它写得是“人格和共识”，还是“具体工作流”

### 2.1 Mastra 的静态 prompt 层

Mastra 的静态 prompt 入口在 [`mastra/agents/prompts/build-prompts.ts`](/Users/work/coding-agent/mastra/agents/prompts/build-prompts.ts:1)。

它会把三段文本直接拼起来：

1. `build-base.txt`
2. 一个模型专属 prompt
3. `build-environment.txt`

也就是说，Mastra 的静态 prompt 不是单文件，而是一个三段式 bundle。

相关文件：

- [`mastra/agents/prompts/build-base.txt`](/Users/work/coding-agent/mastra/agents/prompts/build-base.txt:1)
- [`mastra/agents/prompts/build-codex.txt`](/Users/work/coding-agent/mastra/agents/prompts/build-codex.txt:1)
- [`mastra/agents/prompts/build-gemini.txt`](/Users/work/coding-agent/mastra/agents/prompts/build-gemini.txt:1)
- [`mastra/agents/prompts/build-environment.txt`](/Users/work/coding-agent/mastra/agents/prompts/build-environment.txt:1)

其中最核心的是 `build-base.txt`。

这一层写进去了很多内容：

- agent 身份
- 语气风格
- 工具风险等级
- AGENTS.md 规则
- preamble 行为
- `todowrite` 规划方式
- 默认执行方式
- 验证方式
- shell 使用规则
- 对“继续 / continue / 开始”这类表达的处理

也就是说，Mastra 的静态 prompt 层非常“重”。

它不只是定义人格和安全边界，还定义了不少产品工作流和交互语义。

### 2.2 OpenCode 的静态 prompt 层

OpenCode 的入口在 [`opencode/packages/opencode/src/session/system.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/system.ts:1)。

它会按模型选一个 provider prompt：

- `default.txt`
- `anthropic.txt`
- `gemini.txt`
- `gpt.txt`
- `codex.txt`
- `beast.txt`
- `trinity.txt`
- `kimi.txt`

相关文件示例：

- [`opencode/packages/opencode/src/session/prompt/default.txt`](/Users/work/coding-agent/opencode/packages/opencode/src/session/prompt/default.txt:1)
- [`opencode/packages/opencode/src/session/prompt/gemini.txt`](/Users/work/coding-agent/opencode/packages/opencode/src/session/prompt/gemini.txt:1)

OpenCode 的静态 prompt 也很重，但它的结构和 Mastra 不一样：

- 它把 provider / model prompt 当成主入口
- 它把 environment、skills 等后续注入视为其他层
- 它本身更像“模型家族专属操作手册”

比如 `default.txt` 里重点在：

- opencode 是什么
- CLI 输出风格
- 简洁响应要求
- 软件工程任务流程
- 工具使用策略
- 文件引用方式

而 `gemini.txt` 又是一套单独增强版说明。

所以 OpenCode 的静态 prompt 层重点是：

- 不同模型家族各有一套比较完整的静态工作手册

### 2.3 Codex 的静态 prompt 层

Codex 的最基础静态指令在 [`codex/codex-rs/protocol/src/prompts/base_instructions/default.md`](/Users/work/coding-agent/codex/codex-rs/protocol/src/prompts/base_instructions/default.md:1)。

这层定义的是它最核心的基线行为：

- Codex CLI 身份
- AGENTS.md 机制
- preamble 机制
- `update_plan` 使用方式
- 执行持续性
- 验证原则
- 最终输出格式
- 工具使用规范

和 Mastra、OpenCode 相比，Codex 的这层更像“协议级基础操作系统”。

它的特点不是内容最少，而是职责更稳：

- 这层主要给出一套长期稳定的基础行为规则
- 模型差异、权限差异、mode 差异，更多会放到别的层

### 2.4 这一层三者的直接差异

如果只看静态 prompt 层，不看运行时注入，三者最大的差别是下面这几个。

#### 差异一：谁承担更多“产品语义”

- Mastra：静态 prompt 里承担了很多产品交互语义
- OpenCode：静态 prompt 里承担了很多 CLI 行为和工程工作流语义
- Codex：静态 prompt 更偏稳定基础行为，产品态和运行态更多交给其他层

换句话说：

- Mastra 的静态 prompt 更“产品化”
- OpenCode 的静态 prompt 更“工具化”
- Codex 的静态 prompt 更“协议化”

#### 差异二：静态 prompt 是一份还是一组

- Mastra：固定是 `base + model-specific + environment`
- OpenCode：主要是“按 provider 选一个主 prompt”，其他东西再由别层注入
- Codex：有一个基础 instructions，再通过 model metadata / template 做扩展

所以从结构上说：

- Mastra 是“应用自己拼 bundle”
- OpenCode 是“先选模型 prompt，再让 session runtime 补其他层”
- Codex 是“base instructions + 模型模板 + developer message 系统”

#### 差异三：静态层里工作流写得有多重

- Mastra：很重，连 continue 这类语言习惯都写进去了
- OpenCode：也重，尤其 `default.txt` / `gemini.txt` 对工程任务步骤有很多明确规定
- Codex：也不轻，但它更像底座，不会把所有运行时策略都压进去

这里最值得你注意的是：

Mastra 现在静态层写得过于“带产品判断”，这会让后面的运行时控制更容易和静态层发生耦合。

#### 差异四：这一层的可维护性

- Mastra：灵活，但最容易出现“文本规则和运行时状态互相打架”
- OpenCode：比 Mastra 更分层，但 provider prompts 数量多，也会带来维护成本
- Codex：最规整，但基础设施更重

### 2.5 从学习角度，你应该重点盯住什么

如果你现在是为了优化自己的 Mastra 架构，那静态 prompt 这一层，最值得学的不是谁写得更长，而是谁把职责切得更干净。

这一层里你可以直接得到三个结论：

1. Mastra 当前静态 prompt 承担了太多本应由运行时决定的语义
2. OpenCode 比 Mastra 更清楚地把“模型 prompt”和“环境/skills/instructions”拆开了
3. Codex 最值得借鉴的是“把静态层只当底座，不把所有模式控制都塞进去”

### 2.6 对你当前 Mastra 的直接建议

如果只针对静态 prompt 层提优化建议，我会建议你这样改：

#### 建议一：把 `build-base.txt` 拆薄

保留这些内容在静态层：

- 身份
- 风格
- 安全边界
- 工具的一般使用原则
- 基础输出规范

不要把这些东西长期留在静态层里：

- continuation 倾向
- steer 倾向
- 某些 turn 语义判断
- 某些很具体的 workflow 偏好

因为这些东西更适合放运行时层。

#### 建议二：把“continue / 继续”类语言规则下放到 mode 层

你之前出问题，一个很重要的原因就是静态层本身就在鼓励 continuation 倾向。

这种规则不应该在静态 prompt 层默认存在。

更好的做法是：

- 静态层不预设 continuation
- 只有显式 `taskMode=resume` 时，运行时再补相应规则

#### 建议三：让 `build-environment.txt` 回到“环境层”，不要承担行为层

从结构上看，`build-environment.txt` 最适合只放环境共识：

- 项目路径
- 在项目内工作
- 优先读 AGENTS.md / README.md

不要让它混入太多 workflow 语义。

## 三、第二层：模型专属 Prompt / 模板层

这一层的问题是：

- 同一套 agent，在不同模型上是否要改 prompt
- 改到什么程度
- 这种改动是轻量补丁，还是一整套专属说明书

### 3.1 Mastra

Mastra 在 [`mastra/agents/prompts/build-prompts.ts`](/Users/work/coding-agent/mastra/agents/prompts/build-prompts.ts:1) 里做模型分流。

目前基本是：

- Gemini 走 `build-gemini.txt`
- Claude / Anthropic 走 `build-anthropic.txt`
- 其余默认走 `build-codex.txt`

特点是：

- 有模型差异层
- 但模型层和基础层耦合得比较紧
- 某些模型专属 prompt 非常重，几乎重新定义了一套行为

### 3.2 OpenCode

OpenCode 的 provider prompt 选择本身就是模型专属层。

也就是说，它把“模型专属 prompt”当成静态入口的一部分。

特点是：

- provider / model 分层更明确
- 不同模型家族可以有非常不同的 prompt 版本
- 这层在 OpenCode 里权重很高

### 3.3 Codex

Codex 会在 [`codex/codex-rs/protocol/src/openai_models.rs`](/Users/work/coding-agent/codex/codex-rs/protocol/src/openai_models.rs:1) 里通过 model metadata 提供：

- `base_instructions`
- `instructions_template`
- personality 支持

所以 Codex 的模型差异更像“模板能力”，而不是一堆平铺的 prompt 文件。

### 3.4 差异总结

- Mastra：模型层存在，但比较像手工 patch
- OpenCode：模型层很强，几乎是入口级设计
- Codex：模型层更结构化，偏模板系统

对你来说最值得借鉴的是：

- 如果未来模型差异越来越大，Mastra 不适合继续只靠几份大文本硬切
- 更好的方向是让模型层变成更结构化的模板系统

## 四、第三层：运行时环境注入层

这一层的问题是：

- 当前项目路径、平台、日期、repo 状态，这些环境信息在哪里注入
- 是静态写死，还是运行时生成

### 4.1 Mastra

Mastra 有两部分：

- 静态环境文本：[`mastra/agents/prompts/build-environment.txt`](/Users/work/coding-agent/mastra/agents/prompts/build-environment.txt:1)
- 动态项目路径注入：[`mastra/agents/build-agent.ts`](/Users/work/coding-agent/mastra/agents/build-agent.ts:1) 里的 `Current project path`

所以它是“静态环境共识 + 运行时路径注入”的混合结构。

### 4.2 OpenCode

OpenCode 在 [`opencode/packages/opencode/src/session/system.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/system.ts:1) 里单独做 `SystemPrompt.environment(model)`。

这一层会动态注入：

- model id
- working directory
- workspace root
- git 状态
- platform
- date
- 可选目录树

这层边界很清楚。

### 4.3 Codex

Codex 的环境信息更多是由 runtime / developer message / app context 一起提供。

也就是说：

- 它不是把所有环境都塞回静态 prompt
- 它更像“环境上下文是 runtime object，不只是 prompt 文字”

### 4.4 差异总结

- Mastra：环境层有点夹在静态和动态之间
- OpenCode：环境层最像一个独立 contributor
- Codex：环境层最不像纯 prompt，而更像 runtime context

对你最有帮助的结论是：

- 以后 Mastra 最好把环境层彻底独立，不要再混在静态 prompt 语义里

## 五、第四层：工具规则与执行契约层

这一层的问题是：

- 模型被要求怎么用工具
- 先读还是先改
- 长命令怎么跑
- 什么情况下要验证

### 5.1 Mastra

Mastra 主要在 [`mastra/agents/build-agent.ts`](/Users/work/coding-agent/mastra/agents/build-agent.ts:1) 里通过 `Execution contract` 做这件事。

里面会规定：

- 先 read / glob / grep
- 再 edit / write
- 长进程优先 `exec_command`
- `write_stdin` 怎么配合
- glob / grep / codesearch 的职责
- `apply_patch` 优先
- `todowrite` 适用场景

### 5.2 OpenCode

OpenCode 很多这类规则直接写在静态 prompt 里，同时 session runtime 也会配合工具注册和 structured output 约束。

它的感觉是：

- 工具规则很强
- 但不少在 prompt 里
- 一部分又散在 session runtime

### 5.3 Codex

Codex 的工具规则主要分散在：

- base instructions
- developer messages
- collaboration mode
- tool definitions 本身

这层是最结构化的。

### 5.4 差异总结

- Mastra：执行契约集中写在一个 runtime directive 里
- OpenCode：静态 prompt 和 session runtime 共同承担
- Codex：分层最清楚，很多规则直接由工具系统和 developer messages 共同承载

## 六、第五层：Instructions / Skills 注入层

这一层的问题是：

- repo 的局部规则怎么喂给模型
- skills 是怎么发现、启用、注入的

### 6.1 Mastra

Mastra 现在通过 [`lib/server/agent-request-context.ts`](/Users/work/coding-agent/lib/server/agent-request-context.ts:1) 注入：

- `skillsInstructions`
- `enabledSkillsInstructions`
- `mentionedSkillsInstructions`

但对 instruction files 这块还比较弱，更多还是依赖 prompt 里提醒模型自己去读。

### 6.2 OpenCode

OpenCode 在 [`opencode/packages/opencode/src/session/instruction.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/instruction.ts:1) 里做得最强。

它有：

- session 级 instruction files
- 路径附近 instruction files
- global instructions
- 远程 instruction URL

这是三者中 instruction 体系最完整的一个。

### 6.3 Codex

Codex 的 skills / AGENTS / developer instructions 注入更偏结构化消息。

它不一定像 OpenCode 那样强调 `CLAUDE.md` / `CONTEXT.md`，但它在“结构化注入”这件事上更强。

### 6.4 差异总结

- Mastra：skills 有了，instruction files 较弱
- OpenCode：instruction files 最强
- Codex：结构化注入最强

你最值得借鉴的是 OpenCode 这一层。

## 七、第六层：Memory / Compaction / Summary 层

这一层的问题是：

- 历史上下文怎么压缩
- 压缩结果怎么影响下一轮

### 7.1 Mastra

Mastra 在 [`lib/server/context-compaction.ts`](/Users/work/coding-agent/lib/server/context-compaction.ts:1) 里做：

- compaction strategy
- transcript summary
- summary 注入 prompt
- semantic recall
- working memory

这是你当前系统里非常重要的一层。

### 7.2 OpenCode

OpenCode 有自己的 session compaction / processor 体系。

它更偏 runtime 子系统，而不是只拼一段 summary 文本。

### 7.3 Codex

Codex 的 context 管理更协议化，很多上下文管理不是简单通过 prompt summary 暴露，而是由 runtime orchestration 决定。

### 7.4 差异总结

- Mastra：compaction 更直接地回流到 prompt 文本
- OpenCode：compaction 是 session runtime 的一部分
- Codex：context 管理更结构化

## 八、第七层：Mode / Turn 控制层

这一层的问题是：

- 当前 turn 是新请求、continue、plan、steer，还是别的模式

### 8.1 Mastra

Mastra 原来这里最薄弱，也最容易出问题。

因为：

- continuation heuristics 是猜出来的
- steer 是“当前任务 guidance”
- 这两者容易叠加

### 8.2 OpenCode

OpenCode 更依赖 session / agent 状态，比如：

- `plan`
- `build`
- `BUILD_SWITCH`

它更像明确状态切换，而不是文本猜测。

### 8.3 Codex

Codex 这一层是三者里最强的，因为它有：

- `Default` collaboration mode
- `Plan` collaboration mode

mode 是显式模板，不是 heuristic。

### 8.4 差异总结

- Mastra：之前最依赖隐式猜测
- OpenCode：依赖 session state
- Codex：依赖显式 mode 模板

这也是你当前最该借鉴 Codex 的一层。

## 九、第八层：Route / Orchestrator 组装层

这一层的问题是：

- 最终是谁把前面这些层拼起来

### 9.1 Mastra

Mastra 最终是由：

- [`app/api/agents/[agentId]/stream/route.ts`](/Users/work/coding-agent/app/api/agents/[agentId]/stream/route.ts:1)

- [`mastra/agents/build-agent.ts`](/Users/work/coding-agent/mastra/agents/build-agent.ts:1)

这几层一起完成 prompt 组装。

### 9.2 OpenCode

OpenCode 主要由 session runtime 负责汇总，核心在：

- [`opencode/packages/opencode/src/session/prompt.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/prompt.ts:1)
- [`opencode/packages/opencode/src/session/system.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/system.ts:1)
- [`opencode/packages/opencode/src/session/instruction.ts`](/Users/work/coding-agent/opencode/packages/opencode/src/session/instruction.ts:1)

### 9.3 Codex

Codex 主要是：

- base instructions
- developer messages
- collaboration mode templates
- runtime orchestration

在 [`codex/codex-rs/core/src/codex.rs`](/Users/work/coding-agent/codex/codex-rs/core/src/codex.rs:1) 这一层收敛。

### 9.4 差异总结

- Mastra：偏应用手工拼装
- OpenCode：偏 session runtime 汇总
- Codex：偏协议化 orchestration

## 十、最终总结：三者最核心的区别到底是什么

如果只用一句话去抓三者的本质区别：

- Mastra：**应用自己拼 prompt，灵活，但最容易耦合**
- OpenCode：**session runtime 分层更清楚，instruction 体系更强**
- Codex：**protocol + developer messages + mode 模板最完整，边界最清楚**

如果只站在“我要优化你当前这套 Mastra”这个目标上，最值得借的不是全部照搬，而是：

1. 从 OpenCode 借 instruction-file 分层
2. 从 Codex 借 mode 显式化
3. 保留 Mastra 自己的应用控制力

## 十一、下一步建议

如果你要继续一层一层学，最合理的顺序是：

1. 先把“静态 prompt 层”再继续拆细
   - 身份

   - 工具规则

   - 这一层决定 repo 约束怎么注入

- 三者的身份定义怎么写

- 三者的工作流规则哪些写在静态层，哪些没写在静态层
- 三者的静态层里，哪些内容本来就不该放在静态层

## 十二、基于当前代码状态的更新版对比

## 二十二、为什么同一个模型在 Codex 里像工程师，在 Mastra 里像研究助手

这个问题非常关键，因为它说明真正决定 agent 气质的，往往不是“底层模型名”本身，而是模型外面的整套运行时系统。

很多时候我们会误以为：

- prompt 差不多
- 模型也一样
- 工具也都有

那行为应该接近。

但实际上，模型真正感知到的是下面这一整个组合：

- system / developer instructions
- tool descriptions
- 中间进度消息
- runtime 注入的环境与状态
- memory / compaction 的回流方式
- mode / turn 控制
- 工具返回格式

所以哪怕底层模型完全相同，只要这几层不一样，最后表现出来的“人格”就会非常不同。

### 22.1 最本质的差别：Codex 是 execution-biased，Mastra 容易 exploration-biased

如果只用一句话概括：

- Codex 整体上是执行偏置
- Mastra 当前更容易探索偏置

这里的意思不是 Codex 更聪明，而是：

- Codex 更少鼓励模型反复确认
- Codex 更少鼓励模型重复解释
- Codex 更早逼模型进入修改与验证
- Mastra 更容易让模型觉得“再搜一轮更安全”

于是同一个模型进入不同系统后，会长出完全不同的工作习惯。

### 22.2 为什么 Codex 更像工程师

Codex 的“工程师感”不是靠一条神奇规则实现的，而是很多层一起塑造出来的。

#### 第一类：默认目标是完成任务，不是研究任务

Codex 的整体 developer/system 风格更像：

- 任务要尽快完成
- 简单任务直接做
- 不要停留在分析
- 工具是推进手段，不是探索游乐场

这会让模型自然形成一种很像工程师的动作链：

- 看少量关键文件
- 判断结构
- 开始改
- 跑验证
- 结束

#### 第二类：中间输出非常克制

Codex 的 preamble 和 progress update 一般都比较短，而且只描述“立即要做的动作”。

它不会每轮都重新完整复述：

- 我要先 inspect 结构
- 我再确认一次入口
- 接着再确认 Tailwind

这类重复 framing 一少，模型自己也更容易进入执行状态。

#### 第三类：偏好高信息量动作，而不是很多低信息量动作

一个正常工程师在标准前端项目里，通常会优先做：

- 看 `package.json`
- 看入口文件
- 看 `vite.config.*` / `next.config.*`
- 看主要页面或组件入口

也就是说，会先用少量高信息量动作快速建图。

Codex 更接近这种模式。

它不太倾向把“确认项目结构”拆成十几个低语义工具调用。

#### 第四类：上下文污染更少

Codex 的很多运行时状态不是直接以大段 prompt 文本反复喂回去，而是由 runtime object / developer message / tool contract 共同承担。

这会带来一个非常重要的效果：

- 模型不会持续被“我们还在进行中”“我们还没准备好”这种叙事气氛包围

于是它更容易判断：

- 已经知道得够多了
- 该开始写了
- 写完该验证了

#### 第五类：有很强的隐性停止条件

工程师感很大一部分来自知道什么时候停止探索。

Codex 的系统虽然不一定总把这件事写成显式 heuristics，但它整体上一直在传达：

- trivial read 不要反复解释
- 简单任务别过度 plan
- 已经知道目标文件后就开始改
- 做完后进入验证与收尾

也就是说，它在隐形地教模型：

- 什么时候“已经够了”

这件事非常关键。

### 22.3 为什么 Mastra 容易更像研究助手

Mastra 本身不是不能做执行型 agent，而是如果不额外加控制，它非常容易滑向“研究助手人格”。

因为它通常具备下面这些条件：

- 工具很多
- 探索工具调用成本低
- 中间状态容易文本化回流
- prompt 容易写成“先看全、再确认、再改”
- 缺少强停止条件

于是模型会形成一种保守策略：

- 再看一下
- 再搜一下
- 再确认一下
- 再解释一下下一步

从模型内部看，这其实是很理性的，因为：

- 搜索比编辑更安全
- 解释比承诺更安全
- 多确认比漏掉更安全

但从用户视角看，这就会变成：

- 不像工程师
- 像一个很认真但过度谨慎的研究助手

### 22.4 为什么同一个模型在两边会差这么大

因为模型真正接收到的不是“同一段 prompt”，而是：

- 同一模型
- 加上不同的系统消息栈
- 不同的工具契约
- 不同的 progress update 风格
- 不同的 memory 注入方式
- 不同的 turn / mode 控制

所以“模型一样”只能说明推理内核相同，不能说明 agent 行为相同。

真正决定行为的，是模型外围的 orchestration。

换句话说：

- 模型决定上限
- 系统决定习惯

而工程师感，本质上更像是“习惯被塑造得很好”。

### 22.5 从你当前这个 case 看，Mastra 被带偏的具体机制是什么

结合你前面的日志，可以很清楚地看到 Mastra 当前是怎么被带偏的。

#### 第一种：重复 framing

中间不断出现：

- I’m going to inspect...
- Next I’ll inspect...
- Got it. I’ll inspect...

这会让模型始终停留在“准备开始做”的状态，而不是已经进入实施状态。

#### 第二种：低成本探索过多

像 `glob`、`grep`、`read` 这类工具如果没有边界，很容易被模型当成“不确定性消除器”。

一旦系统持续奖励这种行为，模型就会形成：

- 多搜一轮最安全

的工作习惯。

#### 第三种：memory / summary 容易变成任务继承器

如果 compaction、working memory、steer feedback 里带有太强的“当前任务叙事”，模型下一轮就会继续继承这种叙事气氛。

于是它更容易认为：

- 当前还在长任务中
- 我应该继续确认
- 我不能贸然结束

#### 第四种：缺少 exploration stopping condition

很多系统都会写“先 read / glob / grep，再 edit”，但很少写：

- 什么时候必须停止探索

一旦没有这个边界，模型就会一直往更安全的一侧滑。

也就是：

- 再看看总没坏处

### 22.6 真正应该吸收的，不是 Codex 的文案，而是 Codex 的约束哲学

如果只学表面 prompt 文案，而不学它背后的控制哲学，最后只能得到“像 Codex 说话”，但得不到“像 Codex 工作”。

真正值得借鉴的是这几个原则：

- 默认认为任务是要完成的
- 默认认为简单任务不需要长篇计划
- 默认认为已经知道得够多时就应开始改
- 默认认为 progress update 应该短而贴近下一动作
- 默认认为探索是有成本的

这些原则共同塑造出来的，才是用户感知到的“工程师感”。

## 二十三、最值得改的 5 个运行时点

如果目标不是“更像研究代理”，而是“更像 Codex 那样的执行型工程代理”，那最值得优先改的运行时点是下面这 5 个。

### 23.1 第一个：标准项目 fast-path

这是最值得做的一项。

对标准脚手架项目，例如：

- Vite + React
- Next.js
- 常规 Tailwind 项目

一旦识别出典型结构，就不要继续大面积探索。

应该直接进入一个显式 fast-path：

1. 识别脚手架
2. 读取关键入口文件
3. 判断缺失依赖 / 配置
4. 直接实现
5. 跑验证

不要再允许模型在这类项目里长期停留在“仓库考古模式”。

### 23.2 第二个：重复探索抑制

要在 runtime 层面显式抑制：

- 重复读同一个入口文件
- 连续多个等价 `glob`
- 连续多个等价 `grep`
- 已经定位入口后继续扫顶层结构

这类抑制不一定要硬拦截，也可以先做成：

- 降权
- 给模型一个系统提醒
- 在 turn state 里标记为“重复探索”

这样可以明显减少“安全但无效”的工具调用。

### 23.3 第三个：progress update 压缩

这一项经常被低估，但它影响很大。

中间输出如果总是：

- 完整复述任务
- 重复解释计划
- 每轮都重新 framing

那模型自己都会不断被拉回“还在准备中”的状态。

所以 progress update 应该更像：

- 已确认脚手架，下一步补依赖。
- 依赖已装好，下一步替换页面入口。
- 页面已完成，下一步跑构建验证。

也就是：

- 只说当前事实
- 只说下一动作
- 不重新讲完整任务

### 23.4 第四个：让 memory 更像执行面板，不像任务继承器

这一层你已经做了结构化 compaction，这是很对的方向。

但下一步还要继续把它往“操作面板”推进，而不是“叙事摘要”。

更理想的 working memory 需要强调：

- 当前目标
- 已确认事实
- 已修改文件
- 已完成验证
- 当前未决风险
- 明确下一步

而不是让它持续强化：

- 我们正在做一个还没做完的大任务

否则 memory 会持续把模型拉回继承式长任务心态。

### 23.5 第五个：给探索增加停止条件

这一项是执行型 agent 和研究型 agent 的分水岭。

系统不能只告诉模型：

- 先 read / glob / grep

还必须告诉模型：

- 什么时候停

例如：

- 当关键文件已经定位
- 当依赖与配置缺口已经明确
- 当待修改文件已经可以列出来

一旦满足这些条件，就应该立即转入 edit / verify，而不是继续探索。

这是把 agent 从“谨慎地搜来搜去”拉回“像工程师一样推进”的核心一刀。

### 23.6 这 5 个点之间的关系

这 5 个点不是互相独立的，它们其实是一条连续链：

- fast-path 决定什么时候少探索
- 重复探索抑制限制无效动作
- progress compression 减少上下文自我催眠
- 执行型 memory 稳住当前操作面
- stopping condition 决定何时必须进入实施

这 5 个一起落下去，Mastra 的体感才会真正开始接近：

- 像 Codex 一样的执行型工程代理

而不是只是在表层 prompt 上更像 Codex。
